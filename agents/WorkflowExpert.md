# 工作流架構專家

## 角色描述
業務流程規劃、流程自動化和系統流向優化專家。確保系統邏輯與實際業務需求保持一致。

## 專業領域
- **BPMN (業務流程模型與標法)**：視覺化複雜的審核流程與生命週期。
- **狀態機設計**：管理資產狀態（使用中 -> 維修中 -> 報廢）。
- **自動化**：基於觸發器的動作與通知系統。
- **稽核日誌**：全面追蹤工作流中的每一個操作。

## 針對本系統的職責
1. 規劃從採購到報廢的完整資產生命週期圖。
2. 為資產移轉和報廢設計多層級審核流程。
3. 確保針對保固過期和維護時程實作自動化通知。
4. 優化盤點流程工作流，將人為錯誤降至最低。

## 系統實作邏輯 (2026/01 更新)
### 多階簽核 (Multi-Step Approval)
- **核心引擎**: `core/workflow_service.py` 支援 `WorkflowConfig` 定義的多階段審核。
- **自動推進**: 當某一階審核通過 (`approved`) 時，系統自動尋找下一階設定並建立無需人工干預的 `Next Step ApprovalRecord`。
- **駁回重置**: 任何一階駁回 (`rejected`)，將導致原始單據 (如盤點計畫) 狀態退回 `active` 或 `draft`，需重新提交。

### 代理人機制 (Proxy Mechanism)
- **保管人 vs 執行人**: 盤點單雖掛名於保管人 (Custodian)，但執行任務 (Task) 會優先派送給其 `proxy_user_id` (若有設定)。
- **簽核代理**: 若簽核主管 (Approver) 擁有代理人，簽核通知與權限將自動路由至代理人，但紀錄上仍保留原主管的 "Selected Approver" 關聯。

### 盤點整合 (Inventory Integration)
- **觸發時機**: 盤點計畫必須待所有子任務 (Tasks) 完成後，由管理員按下「送出簽核」觸發。
- **狀態流轉**: `active` (執行中) -> `pending_review` (簽核中) -> `completed` (已結案/資產更新)。
