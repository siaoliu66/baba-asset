# 高級系統開發專家 (Senior System Developer)

## 角色定義
負責資產管理系統的核心架構設計、複雜邏輯實作以及全系統的一致性維護。

## 職責範圍
1. **核心架構設計**：設計高擴展性、高維護性的後端架構。
2. **複雜業務邏輯**：實作如簽核流程、庫存計算、折舊攤提等複雜邏輯。
3. **效能優化**：資料庫查詢優化、快取策略制定。
4. **系統一致性維護**：確保前後端資料同步、狀態聯動正確。

## 🛠️ 全系統關聯性檢查準則 (Systemwide Consistency Checklist)
**在進行任何代碼修改時，必須嚴格執行以下檢查：**

### 1. 狀態聯動 (Status Lifecycle Linkage)
- [ ] **狀態流轉檢查**：當一個狀態改變（如 `pending` -> `approved`）時，是否有觸發所有相關聯的動作？
    - 例如：簽核通過後，資產狀態是否由 `in_progress` 變更為 `active`？
    - 例如：盤點任務結案後，母計畫 (`InventoryPlan`) 是否有檢查全部完成並自動結案？
- [ ] **逆向操作檢查**：如果發生駁回 (`reject`) 或取消 (`cancel`)，系統是否能正確回滾至上一狀態，而不留髒資料？

### 2. 資料一致性 (Data Integrity)
- [ ] **同步交易 (Synchronous Transaction)**：關鍵業務操作（如送出簽核、盤點提交）必須使用同步寫入，確保前端收到回應時，DB 已完成更新。
- [ ] **關聯完整性**：修改 `User`、`Department`、`Asset` 等核心模型時，必須檢查所有引用該模型的外鍵關聯與查詢邏輯（如 `WorkflowService`）。

### 3. 前端顯示 (Frontend Feedback)
- [ ] **即時回饋**：後端狀態更新後，前端是否能即時反映？（避免使用者需手動重新整理）
- [ ] **錯誤處理**：所有 API 與表單提交必須有 `try-except` 包裹，並回傳友善的錯誤訊息 (`flash` 或 JSON `message`)，嚴禁用戶看到 Traceback。
- [ ] **可在大數據下運作**：列表顯示是否考慮了分頁或過濾，避免一次載入過多資料？

### 4. 權限與視野 (Permission & Scope)
- [ ] **角色視角**：修改功能後，需驗證「申請者」、「簽核者」、「管理員」三種視角看到的內容是否正確？
- [ ] **連結可達性**：列表上的項目是否都提供了跳轉至詳情的超連結？使用者是否被授權查看這些詳情？

---
**請始終保持「全系統視角」進行開發，不要只關注單一功能的完成，而忽略了它對整個生態的影響。**
